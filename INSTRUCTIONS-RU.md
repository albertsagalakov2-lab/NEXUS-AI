# NeiroPeiro — что делать после получения архива

## 1. Сначала сохраните старую версию

Самый безопасный вариант — не заменять рабочий сайт сразу.

### Через GitHub

1. Откройте текущий репозиторий.
2. Создайте резервную ветку, например `backup-before-neiropeiro`.
3. После этого загрузите обновлённые файлы в отдельную ветку, например `neiropeiro-redesign`.
4. Проверьте Preview Deployment в Vercel.
5. Только после проверки объедините ветку с `main`.

## 2. Загрузка обновлённого проекта

### Вариант A — новый репозиторий для проверки

Это самый простой и безопасный способ.

1. Создайте на GitHub новый пустой репозиторий.
2. Распакуйте архив.
3. Загрузите содержимое папки проекта в новый репозиторий.
4. В Vercel нажмите **Add New → Project** и импортируйте новый репозиторий.
5. Скопируйте переменные окружения из старого Vercel-проекта.
6. Выполните тестовый деплой.

### Вариант B — существующий репозиторий

1. Создайте новую ветку `neiropeiro-redesign`.
2. Замените файлы содержимым обновлённого архива.
3. Сделайте commit и push.
4. Откройте Preview Deployment в Vercel.
5. Проверьте авторизацию, чат, генерацию фото и генерацию видео.
6. После проверки создайте Pull Request в `main`.

## 3. Переменные окружения

Базу Supabase создавать заново не нужно. В новом Vercel-проекте нужно лишь добавить те же переменные:

```env
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
OPENAI_API_KEY=
OPENAI_API_BASE_URL=
OPENAI_MODEL=
```

Значения берутся из старого проекта Vercel: **Settings → Environment Variables**.

## 4. Локальная проверка

```bash
pnpm install
pnpm dev
```

Откройте `http://localhost:3000`.

Проверка TypeScript и production-сборки:

```bash
pnpm typecheck
pnpm build
```

## 5. Что проверить перед публикацией

- вход и регистрация;
- открытие старых чатов;
- создание нового чата;
- отправка сообщения AI;
- страницы генерации фото и видео;
- профиль и тарифы;
- мобильная версия;
- переменные окружения в Vercel.

## 6. Как откатиться

### Если изменения ещё не объединены с `main`

Просто не объединяйте Pull Request и продолжайте использовать старую ветку `main`.

### Если изменения уже попали в `main`

В GitHub откройте Pull Request с редизайном и нажмите **Revert**. GitHub создаст обратный commit.

Либо в Vercel:

1. Откройте **Deployments**.
2. Найдите последний рабочий деплой старой версии.
3. Откройте его меню.
4. Выберите **Promote to Production**.

База данных при этом не удаляется и не пересоздаётся.
